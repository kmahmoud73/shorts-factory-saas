Governance Layer (NEW — Apr 9, 2026) 1 Module — Sandy-CEO autonomous agent: audit, rule-based remediation, LLM brain, conversational --ask

The Governance Layer sits above everything else. It monitors the other 6 layers, detects failures in real time, auto-remediates known problems, reasons about novel problems via LLM, emails digest reports, and answers natural-language questions from the operator. It is the "CEO" of the factory — what makes Shorts Factory truly autonomous.

Module / ComponentPurposeKey DetailsSaaS Impact
channel_director.py
GOVERNANCE CORE		 4-tier autonomous CEO agent monitoring all 6 channels Tier 1 — Audit: Every 2 hours (7am-11pm Amman, 9 runs/day), walks all 6 channels and computes queue size, upload queue size, upload gap, today's-build status, recent alerts. Output: daily_directive.json.

Tier 2 — Rule-Based Remediation: Maps alerts to concrete subprocess spawns (empty upload queue → spawn builder, low story queue → spawn writer, high upload gap → force uploader). Per-action cooldowns tracked in .director_remediation_log.json to prevent infinite retry loops. All spawns are subprocess.Popen(..., start_new_session=True) — fully detached.

Tier 3 — LLM CEO Brain (_ceo_escalate): When alerts exceed what rules covered, calls Claude Sonnet with a CEO system prompt + full state snapshot. Returns JSON decisions picked from a sandboxed whitelist (_CEO_ALLOWED_ACTIONS + _CEO_ALLOWED_SCRIPTS). Novel reasoning: on breakouts, CEO may spawn auto_scout.py to amplify; on persistent failures, escalates to human via email.

Tier 4 — Conversational --ask: python3 channel_director.py --ask "your question" — pipes factory state into the LLM with a CEO system prompt and returns a grounded briefing with specific CLI commands. Every Q&A logged to .ceo_decisions.json.

Email digest: After every run, consolidated alerts + breakouts + remediations + channel snapshot table emailed via upload_watchdog._send_alert. Section 2

Visual Platform Blueprint

Wireframe-level design for the SaaS control plane: real-time pipeline visualization, decision audit trail, and configuration management.

1. Process Visualization — Real-Time Workflow Canvas

Pipeline Overview Screen

Visual canvas showing all active pipelines as a node graph (similar to n8n or Retool Workflows). Each node is a module. Connections show data flow. Nodes pulse green when running, amber when queued, red on failure.

  • Left panel: channel selector (switch between tenant channels)
  • Center: live node graph with animated data particles on active connections
  • Right panel: selected node detail (last run time, status, input/output preview)
  • Bottom bar: timeline scrubber showing today's schedule with completed/pending/failed markers

Build Monitor

Real-time video build progress. Shows current scene being rendered, which engine is active (Runway/Minimax/Ken Burns), TTS progress, and estimated completion time.

  • Scene-by-scene progress bar with engine badges per scene
  • Live thumbnail preview of last rendered frame
  • Fallback chain indicator (shows which tier each scene used)
  • Cost tracker: per-build cost based on API calls made

2. Decision Visibility — Audit Trail Dashboard

Decision Log

Open Question: Compute Isolation

Video rendering is CPU/GPU-intensive (moviepy + ffmpeg). Options: (A) Shared worker pool with tenant-scoped job queues and fair scheduling, (B) Dedicated container per tenant for builds, (C) Serverless GPU functions (e.g., Modal, RunPod). Option A is cheapest but risks noisy neighbor. Option B is most isolated but expensive. Option C is pay-per-use but adds cold start latency. Recommendation: Start with A, offer B as Enterprise tier upgrade.

Open Question: Claude CLI Dependency

The current llm_client.py has a hardcoded path to the Claude CLI binary (/Users/digisov/.local/bin/claude) as Tier 1 LLM. This works on the owner's machine (free via Claude Pro subscription) but cannot be replicated for tenants in a cloud environment. SaaS must remove the CLI tier and use API-only approach: Anthropic API (primary) with Groq (free fallback). This changes the cost model from $0/day to approximately $0.20-0.50/day per active tenant.

Open Question: YouTube OAuth at Scale

YouTube requires per-channel OAuth consent. Each tenant's channel needs a one-time OAuth flow (browser-based). Options: (A) Platform acts as the OAuth app (single client_id, all tenants authorize through it), (B) Tenants bring their own Google Cloud project. Option A is simpler but puts all tenants under one quota bucket and one Google Cloud project's ToS. Option B is more isolated but increases onboarding friction. Current system uses dual-client approach (separate Google Cloud projects per channel) which maps to Option B.

Risk Register

YouTube ToS Compliance at Scale

HIGH RISK

YouTube's Terms of Service prohibit automated uploads that "mislead users about the purpose of the content." Running 50+ channels with the same pipeline architecture could trigger platform-level enforcement if YouTube detects pattern similarity across channels.

Mitigation: (1) AI disclosure is already baked in (containsSyntheticMedia: true). (2) Each tenant has unique brand, voice, visual style, and content niche. (3) Upload jitter already randomizes timing. (4) Rate-limit total platform uploads to avoid triggering YouTube's bot detection. (5) Legal review of YouTube API ToS for commercial automation use.

API Cost Explosion

HIGH RISK

Current pipeline runs at $0/day via Claude CLI free tier. SaaS replaces this with paid API calls. At 10 tenants, each producing 1 video/day with 5 scenes: ~50 LLM calls/day (scout + write + compliance + CTR + comment replies) + 50 image generations + 50 video animations = potentially $50-100/day in API costs.

Mitigation: (1) BYOK model: tenants bring their own API keys for paid engines. (2) Platform provides free fallbacks only (Pollinations, edge-tts, Ken Burns, Groq). (3) Tiered pricing reflects actual API costs. (4) Implement per-tenant usage caps and metering. (5) Negotiate volume discounts with API providers.

YouTube API Quota Sharing

MEDIUM RISK

YouTube Data API has a 10,000 unit daily quota per Google Cloud project. If the platform uses a single project for all tenants, quota exhaustion would affect all clients. Research alone uses ~860 units/run, uploads ~1,600 units each.

Mitigation: Require each tenant to provide their own Google Cloud project (current dual-client pattern). This isolates quota per tenant. Onboarding wizard includes step-by-step Google Cloud setup.

Community Poster Browser Automation Fragility

MEDIUM RISK

community_poster.py relies on undetected-chromedriver to automate YouTube Studio (because the API does not support community posts). This is fragile: Chrome updates break detection bypass, Google session cookies expire, YouTube Studio UI changes break selectors.

Mitigation: (1) Use cloud browser service (Browserless.io, $50/mo) for reliable headless Chrome. (2) Make community posting optional/degraded gracefully. (3) Monitor for YouTube API community post support (deprecated in 2020 but may return). (4) Per-tenant browser containers for session isolation.

Video Build Performance at Scale

MEDIUM RISK

Each video build takes 5-20 minutes (TTS + image gen + animation + composition). On a single machine, parallel builds can cause CPU contention, TTS collisions (already fixed with story_id prefix), and memory pressure (moviepy/ffmpeg is RAM-hungry).

Mitigation: (1) Job queue with concurrency limits (max N builds simultaneously). (2) Cloud GPU instances for builds (Modal, RunPod). (3) Pre-generate assets asynchronously (images overnight, builds early morning). (4) Build caching: don't regenerate TTS/images if story JSON unchanged.

LLM Quality Variance Across Tiers

LOW RISK

Groq/Llama 3.3 (free fallback) produces noticeably different story quality than Claude Sonnet (Tier 1). Tenants on free fallback may get inferior content quality, leading to complaints or churn.

Mitigation: (1) Clearly communicate LLM tier in pricing (Starter = Groq, Growth = Claude Haiku, Enterprise = Claude Sonnet). (2) Let tenants BYOK for Anthropic/OpenAI. (3) Quality scoring on LLM outputs with auto-retry on low-quality results.

Content Liability

LOW RISK

AI-generated content could contain factual errors, defamatory statements, or copyright issues. The compliance gate catches most issues, but LLM-generated compliance reviews are not legally bulletproof.

Mitigation: (1) Compliance gate is mandatory and non-configurable. (2) Terms of service place content liability on tenant (their channel, their content). (3) Platform provides tools, not guarantees. (4) Optional human review tier for Enterprise clients. (5) Content insurance for platform-managed channels.
